# TDD 가이드

이 문서 커밋 이후 모든 구현은 Red-Green-Refactor 사이클을 준수한다.

## 사이클

1. **Red** — 실패하는 테스트부터 작성
   - 테스트 실행 후 기대대로 실패하는지 확인
   - 커밋 메시지: `Red: <시나리오 설명>`

2. **Green** — 테스트를 통과시키는 최소 구현
   - 과도한 기능/일반화 금지
   - 커밋 메시지: `Green: <구현 설명>`

3. **Refactor** — 중복 제거 및 가독성 개선
   - 리팩토링 동안 모든 테스트가 계속 통과
   - 필요 없으면 생략 가능
   - 커밋 메시지: `Refactor: <정리 내용>`

## 커밋 접두어

| 접두어 | 용도 |
|--------|------|
| `Red:` | 실패하는 테스트 추가 |
| `Green:` | 테스트를 통과시키는 구현 |
| `Refactor:` | 리팩토링 (동작 변경 없음) |
| `docs:` | 문서 변경 |
| `chore:` | 설정·빌드·의존성 변경 |
| `fix:` | 테스트 없이 즉시 고쳐야 하는 긴급 수정 (최소화) |

## 예외 항목

다음은 TDD 사이클 없이 일반 커밋으로 처리한다:
- `application.yaml`, `build.gradle` 등 설정 변경
- README, docs/, 주석
- 단순 오타 수정

## 커버리지

- JaCoCo 전체 커버리지 **80% 이상** 유지
- `./gradlew clean test`에 포함된 `jacocoTestCoverageVerification`이 임계값 강제
- 현재 수치 (main 기준): LINE 91% / BRANCH 81% / INSTRUCTION 90%

## 배경

이 문서 추가 이전 커밋들은 TDD를 엄격히 따르지 않았다 (버그 수정 중심의 즉시 구현 → 커버리지 맞추기 순). 이력을 재작성하는 대신 여기를 기준점으로 삼아 이후부터 Red-Green-Refactor를 준수한다.
